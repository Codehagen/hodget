## Why this matters

Three dependency decisions are currently made by nobody, and each has a small
ongoing cost: (1) a `QueryClientProvider` wraps every route while zero code
uses TanStack Query — dead runtime + two shipped deps; (2)
`@tanstack/react-table` is a production dependency whose only consumers are
the internal `/playbook` component gallery, while every real table is
hand-rolled — two divergent table approaches with no decided winner; (3)
`next dev/build` are pinned to `--webpack` with no recorded reason, opting
out of Next 16's default Turbopack and leaving whether it *could* work
unknown.

*wrong* — the decision is remove-until-needed vs. document-as-scaffolding.

## Steps

### Step 1: Remove the unused TanStack Query infrastructure

Find the provider's mount (likely `app/layout.tsx` or a providers file),
remove `<QueryProvider>` from the tree, delete
`components/query-provider.tsx`, and
`pnpm remove @tanstack/react-query @tanstack/react-query-devtools --filter web`.
Leave a one-line note in the providers file:
`// TanStack Query returns with real data fetching (plan 003) — removed as dead scaffolding until then.`

**Verify**: `pnpm turbo run typecheck test --filter=web` → green;
`grep -rn "react-query" apps/web --include='*.ts*'` → no source matches.

### Step 2: Record the table decision

Verify the react-table import surface:
`grep -rln "@tanstack/react-table" apps/web packages/ui`.
Expected: `apps/web/app/playbook/*` and possibly
`packages/ui/src/components/data-table.tsx`.

- If `packages/ui/.../data-table.tsx` uses it: the shared `DataTable`
  primitive IS the sanctioned react-table wrapper; the decision to record is
  "new filterable/sortable tables use `@workspace/ui` `DataTable`;
  hand-rolled `<Table>` stays for static fixture tables." Append exactly that
  (two sentences) to AGENTS.md's conventions section (or create a
  `## Conventions` note if plan 014 hasn't landed — check first).
- If nothing outside `/playbook` uses it: additionally
  `pnpm remove @tanstack/react-table --filter web` if apps/web declares it
  while only ui consumes it through its own dependency.

**Verify**: typecheck + build green; the convention sentence exists
(`grep -n "DataTable" AGENTS.md`).

### Step 3: Probe Turbopack and document the webpack pin

Run `cd apps/web && npx next build` (no `--webpack`). Two outcomes:

- **It fails** (expected — the Workflow DevKit's compiler and/or MDX loader):
  capture the first error line and add a comment in `next.config.ts` above
  the export:
  `// Bundler pinned to webpack: <tool> fails under Turbopack as of Next 16.2 (<one-line error>). Re-probe on Workflow DevKit / Next upgrades.`
  Keep `--webpack` in scripts.
- **It succeeds**: do NOT switch in this plan. Record in the plan report that
  Turbopack builds clean, add the comment saying so, and leave the switch as
  an explicit follow-up decision for the maintainer (dev-server behavior,
  workflow route generation, and MDX all need a manual pass first).

**Verify**: `cd apps/web && pnpm build` (with `--webpack`) still exits 0;
`grep -n "webpack" apps/web/next.config.ts` shows the decision comment.
